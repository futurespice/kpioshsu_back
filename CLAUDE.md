# CLAUDE.md — kpioshsu_back
KPI-система ОшГУ. Django REST Framework + PostgreSQL.
Правила объединяют поведенческие принципы (Karpathy) и проектную специфику.

**Tradeoff:** уклон в сторону осторожности, а не скорости. Для тривиальных задач — суди по ситуации.

---

## 1. Think Before Coding
**Не предполагай. Не скрывай неясность. Озвучивай трейдоффы.**

Перед реализацией:
- Явно формулируй допущения. При неуверенности — спрашивай.
- Если существует несколько интерпретаций — представь их, не выбирай молча.
- Если есть более простой подход — скажи об этом, возражай там, где нужно.
- Если что-то неясно — остановись. Назови, что именно неясно. Спроси.

---

## 2. Simplicity First
**Минимум кода, решающего задачу. Ничего спекулятивного.**

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» или «конфигурируемости», которую не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если написал 200 строк, а могло быть 50 — перепиши.

Тест: сказал бы сеньор «overcomplicated»? Если да — упрости.

---

## 3. Surgical Changes
**Трогай только то, что должен. Убирай только своё.**

При редактировании существующего кода:
- Не «улучшай» соседний код, комментарии или форматирование.
- Не рефакторь то, что не сломано.
- Соответствуй существующему стилю, даже если сделал бы иначе.
- Если заметил посторонний мёртвый код — упомяни, не удаляй.

При создании orphan-кода твоими изменениями:
- Удали импорты/переменные/функции, которые ТВОИ изменения сделали неиспользуемыми.
- Не удаляй уже существующий мёртвый код без явного запроса.

Тест: каждая изменённая строка должна напрямую следовать из запроса пользователя.

## 4. Goal-Driven Execution
**Определяй критерии успеха. Итерируй до проверки.**

Преобразуй задачи в верифицируемые цели:
- «Добавь валидацию» → «Напиши тесты для невалидных входных данных, затем сделай их зелёными»
- «Исправь баг» → «Напиши тест, воспроизводящий баг, затем сделай его зелёным»
- «Рефактори X» → «Убедись, что тесты проходят до и после»

Для многошаговых задач — кратко формулируй план:
```
1. [Шаг] → проверка: [условие]
2. [Шаг] → проверка: [условие]
3. [Шаг] → проверка: [условие]
```

## 6. Project-Specific Rules

### 6.1 Стек и структура
- **Framework:** Django 5.x + Django REST Framework
- **Auth:** `djangorestframework-simplejwt`
- **DB:** PostgreSQL 15+
- **Cache:** Redis (опционально, только для аналитики)
- **Files:** хранилище файлов — S3 / MinIO / local (≤50 МБ, форматы: PDF, DOCX, JPG, PNG)
- **Docs:** Swagger/OpenAPI 3.0 по адресу `/api/docs`
- **Infra:** Docker + Docker Compose

### 6.2 Критический баг во фронтенде — НЕ ИСПРАВЛЯТЬ
Фронтенд хранит access token под ключом `"acces"` (без второй `s`) — это опечатка в коде фронта.
Backend **обязан** возвращать `{ "acces": "...", "refresh": "..." }` — именно с опечаткой.
Менять поведение без согласования с фронтендом ЗАПРЕЩЕНО.

```python
# ✅ ПРАВИЛЬНО — фронтенд найдёт токен
return Response({
    "acces": str(access_token),    # намеренная опечатка, совпадает с API.ts фронта
    "refresh": str(refresh_token),
})

# ❌ НЕПРАВИЛЬНО — фронтенд НЕ найдёт токен, авторизация тихо сломается
return Response({
    "access": str(access_token),   # грамматически верно, но несовместимо с фронтом
    "refresh": str(refresh_token),
})
```

### 6.3 Роли пользователей
Система использует 8 ролей (ENUM, числа 0–7):

```python
class Role(models.IntegerChoices):
    ADMIN           = 0
    RECTOR          = 1
    VICE_RECTOR     = 2
    SCIENCE_DEP     = 3
    STUDENT_AFFAIRS = 4
    DEAN            = 5
    HEAD_OF_DEPT    = 6
    TEACHER         = 7
```

Иерархия постановки задач: `RECTOR → VICE_RECTOR → DEAN → HEAD_OF_DEPT → TEACHER`

### 6.4 Формула расчёта КПЭ
```
КПЭ преподавателя = Σ(значение_показателя × вес_показателя) / 100
КПЭ кафедры       = среднее(КПЭ всех преподавателей кафедры)
КПЭ факультета    = среднее(КПЭ всех кафедр факультета)
КПЭ университета  = среднее(КПЭ всех факультетов)
```

**Критические проверки при расчёте:**
- Если `total_weight == 0` или нет активных показателей — не делить на ноль, возвращать `0.00`
- Если у кафедры нет преподавателей — КПЭ кафедры = `None`, не `0`
- Результат всегда `DECIMAL(5,2)`, диапазон `0.00–100.00`

### 6.5 Валидация весов КПЭ
```python
# ОБЯЗАТЕЛЬНАЯ проверка при создании/обновлении показателя КПЭ
# Сумма весов всех активных показателей НЕ ДОЛЖНА превышать 1.0

def validate_kpi_weight(new_weight, exclude_id=None):
    current_sum = (
        KPI.objects.filter(is_active=True)
                   .exclude(id=exclude_id)
                   .aggregate(total=Sum('weight'))['total'] or 0
    )
    if current_sum + new_weight > 1.0:
        raise ValidationError("Сумма весов показателей превышает 1.0")
```

### 6.6 Soft delete — обязательно для всех моделей
Все сущности используют soft delete. Hard delete запрещён.

```python
# Каждая модель должна иметь:
deleted_at = models.DateTimeField(null=True, blank=True, default=None)

# Менеджер по умолчанию фильтрует удалённые:
objects = SoftDeleteManager()  # возвращает только deleted_at=None
all_objects = models.Manager()  # включая удалённые
```

**Исключение:** `DELETE /api/kpi/{id}` — это деактивация (`is_active=False`), не удаление записи.

### 6.7 Формат ответов API

**Успешный список (пагинация):**
```json
{
  "count": 10,
  "page": 1,
  "page_size": 20,
  "next": "http://...",
  "previous": null,
  "result": []
}
```

**Ошибка:**
```json
{
  "error": "Описание ошибки",
  "code": "ERROR_CODE",
  "details": {}
}
```

Поле называется `"result"` (не `"results"` как в DRF по умолчанию) — кастомный pagination класс обязателен.

### 6.8 Валидация данных
- Email: обязан заканчиваться на `@oshsu.kg`
- Значение КПЭ: целое число `0–100`
- Приоритет задачи: только `"high"`, `"medium"`, `"low"`
- Дедлайн при создании: дата не может быть в прошлом
- Файлы: максимум 50 МБ, форматы PDF / DOCX / JPG / PNG

### 6.9 Нефункциональные требования
- Время ответа API: ≤ 1 сек для 95% запросов
- CORS: разрешить домен фронтенда (из `CORS_ALLOWED_ORIGINS`)
- Rate limiting: ≤ 100 запросов/мин на IP
- Логирование: audit log для всех изменений данных
- Хранить историю расчётов КПЭ за все периоды (не перезаписывать)

### 6.10 Приоритет реализации
**Фаза 1 (Sprint 1–2) — КРИТИЧНАЯ:**
Auth → Структура (Faculty/Department/User) → Tasks (CRUD) → KPI (показатели + расчёт)

**Фаза 2 (Sprint 3–4) — ВАЖНАЯ:**
Publications → Documents/UMK → Approvals → Analytics (Rector/Vice-Rector)

**Фаза 3 (Sprint 5+) — ДОПОЛНИТЕЛЬНАЯ:**
Planerka → Department Load → Dean Analytics → Notifications → Export
